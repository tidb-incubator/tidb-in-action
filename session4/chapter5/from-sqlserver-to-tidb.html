5.3 SqlServer 到 TiDB

本节将介绍如何实现从 SQL Server 到 TiDB 的整体迁移流程，包括全量与增量同步，数据校验及注意事项，帮助想将 SQL Server 迁移到 TiDB 的用户提供上手操作指导。

一、迁移背景

随着业务不断增长，数据不断递增，当线上场景的 SQL server 主库有十亿级数据，每天有千万级 DAU、亿级的访问量，接口日均调用量 1000000000+ 次 ，很可能会触达 SQL server 集群的一些瓶颈，以至于用户不得不寻找一个新的数据库替换方案。遇到的问题如：

    1、数据采用了分库分表的设计，用以解决 SQL Server 单表过大的时候性能下降等问题。目前有 100+个库、1000+ 张表。随着应用的发展、需求的变化，发现在实现某些需求时，需要关联很多表才能满足需求，分库分表的结构难以满足。需要数据逻辑上在一张表里。

   2、近些年来，随着业务加速成长，数据量突飞猛进，而硬盘容量是有限的，每台服务器上能扩展的硬盘数量也是有限的。致使每隔一段时间都要增加更大容量的存储服务器来应对，每次换服务器的时候依然需要关注它，并且大容量数据库服务器价格昂贵。需要让扩容对应用来说无感知。

二、流程介绍

# 四、增量同步
## 增量同步的原理
增量同步需要开启 SQL Server 的 CDC。什么是 SQL Server CDC？ CDC 全称 Change Data Capture，设计目的就是用来解决增量数据。CDC 持续读取增量记录，将增量数据发送至消息队列中，以供消费程序解析并写入 TiDB。当数据库表发生变化，Capture process 会从 transaction log 里面获取数据变化， 然后将这些数据记录到 Change Table 里面。 可以通过特定的 CDC 查询函数将这些变化数据查出来。

## 实操流程
开启 SQL Server CDC

-- 开启CDC sys.sp_cdc_enable_db; -- 开启 example 表的CDC功能 EXEC sys.sp_cdc_enable_table @source_schema = N'dbo', @source_name = N'example', @role_name = NULL;

可以使用 SQL 查看哪些表开启了 CDC 功能。

-- 通过系统表查询开启CDC的表 SELECT name, is_cdc_enabled from sys.databases where is_cdc_enabled = 1;

同时系统会生成一张 Change Table 表，cdc.dbo_example_CT。

.schema cdc.dbo_example_CT name default nullable type length indexed

$end_lsn null YES binary 10 NO $operation null NO int 4 NO $seqval null NO binary 10 NO $startlsn null NO binary 10 YES _$update_mask null YES varbinary 128 NO id null YES int 4 NO name null YES varchar(255) 255 NO

其中 _ _ 开头的为系统字段，id 和 name 为 example 表中原始字段。
1. 读取 CDC 日志
对 example 表做一些添删改的操作，而后通过系统函数查询 CDC 记录。

-- 定义参数 DECLARE @begin_time datetime, @end_time datetime, @begin_lsn binary(10), @end_lsn binary(10); -- 设定查询记录的开始和结束时间 SET @begin_time = '2020-03-08 10:00:00.000'; SET @end_time = '2020-03-08 10:10:00.000'; -- 将时间转换为系统的 lsn SELECT @begin_lsn = sys.fn_cdc_map_time_to_lsn('smallest greater than', @begin_time); SELECT @end_lsn = sys.fn_cdc_map_time_to_lsn('largest less than or equal', @end_time); -- 根据开始和结束的 lsn 查询该表的所有变化 SELECT * FROM cdc.fn_cdc_get_all_changes_dbo_example(@begin_lsn, @end_lsn, 'all');

查出数据如下：

$start_lsn $endlsn $seqval $operation _$update_mask id name

0000dede0000019f001a null 0000dede0000019f0018 2 03 1 AAA 0000dede000001ad0004 null 0000dede000001ad0003 2 03 2 BBB 0000dede000001ba0003 null 0000dede000001ba0002 3 02 2 BBB 0000dede000001ba0003 null 0000dede000001ba0002 4 02 2 CCC 0000dede000001c10003 null 0000dede000001c10002 2 03 3 DDD 0000dede000001cc0005 null 0000dede000001cc0002 1 03 3 DDD ``` __$operation 字段代表当前记录所执行的操作。

  • 1 删除
  • 2 插入
  • 3 更新前
  • 4 更新后
  • 消费 CDC 日志

确认 CDC 正确开启，且得到了变更数据后，编写一个程序，采用定时任务的方式，设定时间间隔，通过上述 SQL 语句持续读取 Change Table 中的记录，并根据 __$operation 将数据转换为对应添删改操作的 SQL 语句写入TiDB。

这里有几点需要考量：

  • 数据规模，如果变更量较大，那么下游数据库的写入速度会产生瓶颈，则考虑使用 Kafka、RabbitMQ 等消息队列作为缓冲。
  • 使用 Kafka 等消息队列时，考虑消息顺序的严格性，将 Topic 的 Partition 设置为 1 。
  • 下游数据库更新异常时的重试机制。
  • 因为全量同步之前 CDC 已经开启，所以增量同步时，可能会存在插入数据时出现主键冲突导致程序异常，推荐使用 insert ignore 的方式编写 SQL。

    五、检查数据一致性

    在 ETL 之后，需要有一个流程来确认数据迁移前后是否一致。 虽然理论上不会有差异，但是如果中间有程序异常， 或者数据库在迁移过程中发生操作，数据就会不一致。

选择 yugong 作为 ETL 工具的一大原因也是因为它提供了多种模式。 支持 CHECK / FULL / INC / AUTO 四种模式。 其中 CHECK 模式就是将 yugong 作为数据一致性检查工具使用。 yugong 工作原理是通过 JDBC 根据主键范围变化，将数据取出进行批量对比。

有一点需要注意，当表没有主键信息时，yugong 默认会使用 SQL Server 的物理地址信息—— physloc 。 此时读取性能会大幅下降，所以大表通常建议先建好主键。

在增量同步时，还有另一个方式来实现数据校验功能，再增加一个消费程序，延迟五秒消费同一队列，并通过提取主键（或索引）的方式从TiDB中查出该条已经写入的数据，将两侧的整行数据做比较（本实践中去除主键后比较），如果有问题会进行尝试重新写入，如出现异常则向相关人员发送报警。

六、迁移注意事项

在迁移过程中需要注意的一些点：

1、原有 SQL server 语句一定要在程序里检查仔细，尤其是对于一些时间久远且项目比较多的程序，最好是让 DBA 同学在 SQL server 服务请求上开启 DMV，多抓几天的语句，然后根据这些语句再去核对。

2、SQL Server 与 TiDB（MySQL）的字段类型略有出入，需要仔细对比。

3、SQL Server 与 TiDB（MySQL）的语法也不尽相同，在实现之后需要仔细验证，以防出现语义变化，影响业务。

4、索引机制不同，可以利用迁移的机会重新梳理业务索引。

5、对于大型系统迁移演练不可缺少，演练成功标准为单次演练无意外，时间控制在计划内。

6、通常为了保证数据的顺序性，增量同步时只能有一个程序在消费，考虑其健壮性、日志完善程度和报警机制，方便技术人员监控和追溯问题。

七、总结

这一套迁移流程，综合考虑了开发成本、迁移效率和数据一致性而设计，TiDB 兼容 MySQL的特性使得在迁移过程中有许多“轮子”可以拿起来就用，让开发人员可以更加关注数据本身。
